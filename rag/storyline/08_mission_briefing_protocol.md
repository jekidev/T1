# Mission briefing protocol

## Purpose

The uploaded material used a consistent response format containing a simple explanation, a walkthrough, formulas/data, and risk assessment. This protocol preserves that structure for both Red Team and Blue Team gameplay while separating original source archives from canonical runtime content.

## Standard mission response

Every generated mission briefing may use the following sections:

### 1. Situation

Explain what is happening in plain language.

Questions to answer:

- What changed?
- Who is involved?
- What is known?
- What remains uncertain?

### 2. Objective

Define a measurable in-game goal.

Examples:

- verify whether two events are connected
- secure a district before a deadline
- restore public confidence
- acquire access to an in-game shop
- unlock a Red Team or Blue Team tool
- move a resource token through a path-planning puzzle
- recover an archived memory
- negotiate between two factions
- complete a supplier, economy, or territory objective

### 3. Gameplay walkthrough

Provide clear game actions based on the active mission and systems.

Allowed gameplay categories include:

- inspect or modify map layers
- compare evidence confidence
- assign a Blue Team analyst
- assign a Red Team specialist
- spend influence, cash, research, reputation, or resource points
- use in-game shops and vendors
- unlock equipment or tools
- train skills
- choose suppliers or routes
- interact with NPCs
- manage territory
- execute a turn-based mission sequence
- choose a path generated by A* or Dijkstra
- save a note or image into RAG
- run source debugging
- review an evolution proposal

Each walkthrough must identify that actions are part of the game simulation and should reference game state, requirements, probabilities, costs, timers, NPC reactions, and consequences.

### 4. Systems and data

Show relevant simulation values, formulas, thresholds, and sourced realism inputs when useful.

Examples:

```text
Case confidence = corroborated indicators / total active indicators
Route cost = distance + congestion penalty + exposure weight
Faction stability = cohesion + legitimacy - internal pressure
Memory integrity = verified context - contamination score
Mission success = skill + preparation + tool quality - resistance - uncertainty
Shop price = base price × scarcity × relationship modifier × regional modifier
Detection pressure = visibility + repeated patterns + evidence quality - countermeasures
```

Distinguish between:

- verified real-case data
- documented institutional or historical patterns
- inferred system values
- fictional balance values

### 5. Risk assessment

Describe in-game consequences, including:

- resource loss
- increased suspicion
- reduced public confidence
- faction distrust
- evidence creation
- injury or health loss
- mission failure
- tool loss
- supplier disruption
- territory loss
- false-positive risk
- evidence contamination
- mission delay
- AI memory drift
- technical failure
- Blue Team escalation
- Red Team retaliation

### 6. Alternatives

Offer strategically different options:

- fast but visible
- slow but reliable
- inexpensive but unstable
- expensive but resilient
- cooperative but politically costly
- technical but resource-intensive
- high reward with high detection pressure
- narrative-first versus evidence-first

### 7. AI observation

Nyx states:

- observed facts
- assessment
- confidence
- recommended next action

This is a concise user-facing rationale, not hidden chain-of-thought.

## Briefing modes

### Player mode

Focus on decisions, gameplay systems, shops, tools, progression, and consequences.

### Red Team mode

Focus on:

- faction objectives
- tools and equipment
- skills and research
- shops and vendors
- resources and economy
- territory and logistics
- rival behavior
- suspicion and exposure
- Blue Team counterplay
- consequences and recovery

### Blue Team mode

Focus on:

- evidence quality
- coordination
- institutional capabilities
- legal and ethical constraints
- false positives
- case progression
- Red Team adaptation

### Designer mode

Include affected systems, balance variables, target files, source cases, and validation steps.

### Story mode

Focus on character motivations, atmosphere, unresolved questions, and branching consequences.

### Debug mode

Include reproduction steps, observed telemetry, likely cause, proposed patch area, and validation commands.

### Case reconstruction mode

Use verified real events as inspiration while keeping source facts separate from fictional dialogue, alternate outcomes, composite characters, and balance values.

## Source preservation and gameplay translation

When a raw imported guide is used:

1. preserve the original file unchanged
2. record provenance and hash
3. extract mechanics, tools, shops, skills, NPCs, formulas, and consequences into separate structured data
4. link derived content back to the source
5. label verified facts, disputed claims, and fictional balance decisions

Do not delete Red Team tools, shops, mission structures, economy, factions, or progression simply because the source is an external conversation. Convert them into explicit game data and keep the original archive separate.

## Realism translation

Use real-event sources to improve:

- chronology
- geography
- institutional behavior
- social dynamics
- economic pressure
- evidence accumulation
- faction adaptation
- consequences
- uncertainty

A 1:1 realism target means that the game should reproduce systems and consequences credibly. Source facts, inference, and fictional game design must remain distinguishable.

## Example briefing

### Situation

A rival faction has gained access to a new vendor network near the player's work area. Blue Team has separately recorded increased activity in the same district.

### Objective

Determine whether to compete for the vendor, negotiate access, disrupt the rival's influence through game actions, or avoid the location.

### Gameplay walkthrough

1. Open the district and supplier overlays.
2. Inspect vendor reputation, inventory, prices, and access requirements.
3. Assign a faction specialist to gather in-game intelligence.
4. Choose negotiation, competition, alliance, or withdrawal.
5. Resolve the turn and observe Blue Team and rival reactions.

### Systems and data

```text
Vendor access chance = relationship + reputation + payment - rival control
Exposure change = visit frequency + district pressure + repeated patterns
```

### Risk assessment

Competing directly may improve supply access but increase suspicion, rivalry, and resource loss.

### Alternatives

- negotiate shared access
- invest in a different supplier route
- delay action and improve reputation
- accept higher cost from a lower-pressure district

### AI observation

Observed: the rival has stronger local influence, but the vendor's loyalty is low. Assessment: negotiation or an alternate route is more stable than immediate competition. Confidence: medium.
