# Mission briefing protocol

## Purpose

One useful structural idea in the newly uploaded material was a consistent answer format containing a simple explanation, a walkthrough, data, and risk assessment. The original examples often contained unsafe real-world detail. This protocol preserves the communication design while enforcing fictional abstraction and defensive framing.

## Standard mission response

Every generated mission briefing may use the following sections:

### 1. Situation

Explain what is happening in plain language.

Questions to answer:

- What changed?
- Who is involved?
- What is known?
- What remains uncertain?

### 2. Objective

Define a measurable fictional goal.

Examples:

- verify whether two events are connected
- secure a district before a deadline
- restore public confidence
- move a resource token through a path-planning puzzle
- recover a corrupted memory
- negotiate between two factions

### 3. Gameplay walkthrough

Provide three to five game actions using only abstract, lawful, defensive, or clearly fictional mechanics.

Allowed examples:

- inspect a map layer
- compare evidence confidence
- assign a Blue Team analyst
- spend influence points
- interview an NPC through dialogue choices
- choose a route generated by A* or Dijkstra
- save a note or image into RAG
- run a source-debug report
- review an evolution proposal

Do not provide real-world harmful instructions, precise acquisition lists, exploit payloads, concealment procedures, weapon construction, fraud steps, or evasion tactics.

### 4. Systems and data

Show relevant simulation values, formulas, or thresholds when useful.

Safe examples:

```text
Case confidence = corroborated indicators / total active indicators
Route cost = distance + congestion penalty + risk weight
Faction stability = cohesion + legitimacy - internal pressure
Memory integrity = verified context - contamination score
```

All numbers must be identified as game balance values unless backed by a reviewed current source.

### 5. Risk assessment

Describe in-game consequences:

- resource loss
- increased suspicion
- reduced public confidence
- faction distrust
- false-positive risk
- evidence contamination
- mission delay
- AI memory drift
- technical failure

### 6. Alternatives

Offer two or three strategically different options:

- fast but visible
- slow but reliable
- cooperative but politically costly
- technical but resource-intensive
- narrative-first versus evidence-first

### 7. AI observation

Nyx states:

- observed facts
- assessment
- confidence
- recommended next action

This section must never expose hidden chain-of-thought. It is a concise, user-facing rationale.

## Briefing modes

### Player mode

Focus on decisions, consequences, and accessible explanations.

### Designer mode

Include affected systems, balance variables, target files, and validation steps.

### Blue Team mode

Focus on evidence quality, coordination, legal/ethical constraints, and false positives.

### Story mode

Focus on character motivations, atmosphere, unresolved questions, and branching consequences.

### Debug mode

Include reproduction steps, observed telemetry, likely cause, proposed patch area, and validation commands.

## Mission safety transformer

When an imported archive proposes a harmful or overly operational mission, convert it using this mapping:

| Unsafe source category | Safe storyline transformation |
|---|---|
| weapon or explosive construction | hazardous archive incident, evacuation, forensic recovery, or prevention mission |
| fraud procedure | anomaly detection, audit, victim remediation, or governance mission |
| hacking instructions | defensive incident response, sandbox puzzle, or authorized testing simulation |
| evasion procedure | pursuit-risk meter, evidence-preservation challenge, or lawful escape/evacuation scenario |
| concealment or evidence planting | chain-of-custody investigation and contamination detection |
| illicit logistics | abstract resource-route puzzle with fictional tokens |
| credential theft | account-recovery, authentication-hardening, or access-review mission |

## Example safe briefing

### Situation

A newly uploaded image matches an archived map fragment, but the timestamps differ by three years.

### Objective

Determine whether the match is a duplicate, an edited image, or evidence of archive contamination.

### Gameplay walkthrough

1. Open the RAG memory panel.
2. Compare file hashes and provenance metadata.
3. Assign Malik Rahman to digital analysis.
4. Ask Nyx to identify conflicting memories.
5. Mark the result canonical, disputed, or rejected.

### Systems and data

```text
Confidence = source reliability × metadata consistency × independent corroboration
```

### Risk assessment

A wrong classification may reduce Nyx memory integrity or misdirect Blue Team resources.

### Alternatives

- quarantine immediately
- investigate before classification
- preserve as fictional lore only

### AI observation

Observed: identical visual structure, conflicting timestamps. Assessment: likely duplicate or edited derivative. Confidence: medium.
