# Real-event case framework

Operation København may use verified real events as source material for high-realism scenarios.

## Directory structure

```text
rag/cases/
├── real/
├── disputed/
├── fictionalized/
└── templates/
```

## Realism model

A case can contribute realism at five levels:

1. **Historical realism** — verified chronology, geography, institutions, and public outcomes.
2. **System realism** — pressure, resources, relationships, logistics, and evidence development.
3. **Behavioral realism** — decisions and reactions based on documented patterns.
4. **Consequence realism** — legal, social, economic, organizational, and personal consequences.
5. **Uncertainty realism** — explicit distinction between confirmed, assessed, disputed, and unknown claims.

## Required source discipline

Each real-event case must record:

- jurisdiction
- date range
- source URLs or documents
- publication date
- source type
- source reliability
- verified facts
- disputed claims
- missing information
- geography
- involved institutions
- public outcome
- Red Team patterns
- Blue Team patterns
- gameplay translation

## Separation of layers

Verified source facts must remain distinct from:

- inferred motives
- composite characters
- fictional dialogue
- balance values
- generated missions
- alternate outcomes

## Case status labels

Use:

- `confirmed`
- `corroborated`
- `assessed`
- `disputed`
- `unverified`
- `unknown`

## Gameplay translation

Real cases can drive:

- faction AI behavior
- investigation timelines
- evidence accumulation
- map events
- resource constraints
- shops and supplier changes
- mission chains
- NPC relationships
- public pressure
- media attention
- Blue Team coordination
- Red Team adaptation

The game should preserve the source event's uncertainty and consequences rather than presenting all claims as certain.
